TOPIC: LRU Cache (Least Recently Used)

Fixed-size cache with O(1) operations using hash map + doubly-linked list

What this carousel covers:
→ O(1) get() and put() operations through hash map + doubly-linked list combination
→ Automatic eviction of least recently used items when capacity is reached
→ Web browser page cache simulation with capacity 3 demonstrating cache hit/miss patterns
→ std::optional usage for safe cache miss handling without exceptions
→ std::list::splice() for O(1) access order tracking (just pointer updates)
→ Cache hit optimization: ~1ns hash lookup + ~1ns list splice vs disk I/O at ~1-10ms

Key insights:
◆ Performance at 90% hit rate = 10× average speedup; 50% hit rate = 2× speedup compared to always loading from disk
◆ Real browsers keep ~100-500 pages in memory using LRU; CDNs like Cloudflare/Akamai use LRU for edge caching
◆ Doubly-linked list required for O(1) removal (node->prev->next = node->next); singly-linked would be O(n)
◆ LRU vs LFU trade-off: LRU evicts oldest accessed (simpler), LFU evicts least frequently accessed (better for frequency patterns)
◆ Memory bounded regardless of total items accessed - 3-page cache uses constant memory even accessing thousands of pages
◆ Database systems use LRU for query result cache, index pages (PostgreSQL shared buffers, MySQL query cache)
◆ Operating systems use LRU for page table cache and disk block cache (buffer cache in Linux kernel)
