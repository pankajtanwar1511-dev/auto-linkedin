TOPIC: poll() - Scalable I/O Multiplexing Without FD Limits

Evolution beyond select() with unlimited file descriptor support

What this carousel covers:
→ select() limitations: FD_SETSIZE cap at 1024, O(max_fd) complexity, expensive fd_set copying
→ poll() API: struct pollfd array, separate events/revents fields, millisecond timeout
→ Event flags: POLLIN/POLLOUT (input), POLLERR/POLLHUP/POLLNVAL (output)
→ Dynamic FD management: iterate backwards for safe removal, free slot reuse with fd=-1
→ POLLOUT monitoring: write queues for partial sends, only enable when data pending
→ Performance: O(num_fds) vs select's O(max_fd), 4-6.7x faster with sparse FDs
→ Production patterns: connection limits (MAX_CLIENTS), idle timeout detection, graceful shutdown

Key insights:
◆ No FD_SETSIZE limit: Can monitor millions of FDs vs select's hardcoded 1024 limit
◆ Sparse FD efficiency: With 10 active from 5000 total, poll() iterates 10 vs select's 5000
◆ API improvements: Array of structs vs opaque bitmasks, no max_fd tracking, no copying
◆ Performance comparison: 100 FDs/10 active takes 10μs vs select's 100μs in dense scenarios
◆ Critical pattern: Iterate backwards (i = size-1; i >= 0; i--) for safe vector erase
◆ Free slot reuse: Set fd=-1 to mark deleted entries, poll() ignores negative FDs
◆ POLLHUP order matters: Check POLLIN first to read final data, then handle POLLHUP
◆ Early exit optimization: Break after processing 'ready' count, avoid full array scan
◆ Write monitoring: Add POLLOUT only when write_queue has data, remove when empty
◆ Timeout modes: -1 blocks forever, 0 non-blocking, N milliseconds for periodic tasks
◆ Production features: Idle client detection (30-60s timeout), MAX_CLIENTS enforcement
◆ Limitations: Still O(n), kernel copies entire array per call, no edge-triggered mode
