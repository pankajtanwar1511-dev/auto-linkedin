TOPIC: C++20 Ranges and Views - Modern Iteration and Transformation

Composable, lazy, and elegant alternatives to traditional STL algorithms.

What this carousel covers:
→ Composable pipelines with pipe operator (|) - chain operations left-to-right like Unix pipes
→ Lazy evaluation - compute elements on-demand only when iterated, not when created
→ Zero intermediate containers - no memory overhead from temporary vectors
→ Range concepts hierarchy (input_range → forward_range → bidirectional_range → random_access_range → contiguous_range)
→ Standard views (filter, transform, take, drop, reverse, join, split, keys, values, iota)
→ Projections - transform elements before applying operations (e.g., sort by &Person::age)
→ Range algorithms with better error messages and useful return values
→ Performance: O(1) view construction vs O(n) container construction
→ Lazy vs eager evaluation - processes only 20 elements instead of 1,000,000 when using take(10)
→ Common patterns: filter + transform + collect, infinite sequences, flattening nested ranges

Key insights:
◆ Pre-C++20 problem: Filtering, transforming, and taking first N elements required 3 intermediate vectors, verbose iterator pairs, and eager evaluation processing all elements
◆ C++20 solution: Chain operations with | operator - zero intermediate containers, lazy computation on-demand
◆ Views are non-owning (reference underlying data), O(1) construction cost, movable, composable with | operator
◆ Lazy evaluation example: Creating view = O(1), iterating with take(3) transforms only 3 elements, not all 10
◆ Performance with huge_vector(1,000,000): Lazy approach processes ~20 iterations to find 10 evens, eager approach processes 1,000,000 iterations + allocates 500,000 elements
◆ Projection example: std::ranges::sort(people, {}, &Person::age) - no custom comparator needed, just project to age field
◆ Range concepts enforce capabilities: forward_list supports forward_range but not bidirectional_range, so views::reverse compilation fails
◆ View composition weakens range categories: vector is contiguous_range, but vector | views::filter becomes bidirectional_range (filter can't maintain contiguity)
◆ Critical edge case: Views are non-owning - returning temporary vector | views::filter causes dangling references (vector destroyed after return)
◆ Multiple iterations recompute: Iterating same view twice calls transform function twice (materialize to vector if iterating multiple times)
