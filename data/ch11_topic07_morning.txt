TOPIC: STL Container Thread Safety - Concurrent Access Rules and Patterns

Understanding the minimal thread safety guarantees of standard containers.

What this carousel covers:
→ The golden rule: multiple const reads are safe (C++11 §23.2.2), but all writes require synchronization
→ Why operator[] on maps is dangerous - it's a WRITE operation that inserts default values
→ Container-specific risks: vector reallocation vs map rebalancing vs unordered_map global rehashing
→ Iterator invalidation in concurrent contexts - TOCTOU races and happens-before relationships
→ Why std::vector<bool> is unsafe even for different indices (bit-packing causes false sharing)
→ The copy-on-write problem that got banned in C++11 (reference counts modified during reads)
→ Load factor and rehashing - why reserve() is mandatory for unordered_map thread safety
→ Production patterns: thread-local accumulation, shared_mutex for read-heavy workloads, and wrapper classes

Key insights:
◆ STL containers are NOT thread-safe for writes - concurrent push_back() causes data races (lost writes, wrong size, or crashes during reallocation)
◆ Use at() or find() for read-only map access, never operator[] - it inserts missing keys and may trigger rehashing
◆ std::unordered_map rehashing invalidates ALL iterators globally, while std::map only affects local tree paths - always call reserve(2x expected size)
◆ Even "read-only" operations on non-const containers aren't guaranteed thread-safe - the standard only protects const-qualified concurrent reads
◆ std::vector<bool> packs bits into bytes - writing to flags[0] and flags[1] races on the same byte
◆ Thread-local accumulation with single merge beats locking every push_back by 10-100x - batch operations to minimize contention
◆ For read-heavy workloads (10:1+ ratio), shared_mutex with shared_lock allows concurrent readers while writers get exclusive access
◆ Iterator lifetime must be protected by locks in multithreaded code - any thread can trigger reallocation and invalidate all iterators at any time
