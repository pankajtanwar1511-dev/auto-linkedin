TOPIC: Copy-on-Write (COW) String

Share data until modification, then copy

What this carousel covers:
→ Reference counting mechanism with atomic<int> refcount tracking shared buffers
→ Lazy duplication - copy constructor is O(1) just incrementing refcount to 2
→ Detach-on-write trigger when operator[] non-const checks refcount > 1
→ Buffer structure with flexible array member char data[1]
→ Memory allocation using ::operator new(sizeof(Buffer) + len)
→ Thread-safe atomic operations: fetch_add, fetch_sub with memory_order_relaxed/release/acquire
→ Iterator invalidation scenarios when detach creates new buffer

Key insights:
◆ Copy constructor is O(1) constant-time vs O(n) - just atomic refcount increment
◆ 1000 copies of same string share 1 buffer instead of 1000 separate allocations
◆ Detach cost is O(n) allocation + memcpy but amortized if many reads precede it
◆ Atomic refcount operations add ~10-20 CPU cycles overhead per access
◆ std::string stopped using COW in C++11 - incompatible with move semantics
◆ Modern std::string uses SSO (Small String Optimization) instead for small strings stored inline
◆ COW was valuable because most string operations are read-only in typical code
◆ Thread-safety overhead and unpredictable detach behavior made COW obsolete
