TOPIC: std::list - Doubly Linked List Container

Master doubly linked lists with O(1) insertion/deletion and iterator stability, understanding when scattered memory and pointer overhead outweigh algorithmic benefits.

What this carousel covers:
→ Node structure - each element has 16 bytes overhead (prev and next pointers on 64-bit)
→ Non-contiguous memory - nodes scattered in heap, poor cache locality
→ Iterator stability - only erased element's iterator invalidates, all others remain valid
→ O(1) operations with iterator - insert/erase at position when you have iterator
→ Splice operation - O(1) node transfer between lists, no allocation/copying
→ Member functions vs algorithms - list::sort() required, std::sort won't compile
→ Cache performance reality - vector 10-100x faster iteration despite same O(n)
→ Memory overhead - 5x vector for int (20 bytes vs 4 bytes per element)
→ Real-world LRU cache - combines splice for O(1) reordering with hash map lookup

Key insights:
◆ Iterator stability is list's killer feature - modifications don't invalidate other iterators
◆ Splice transfers ownership without copying - just pointer rewiring, O(1) for single elements
◆ No operator[] random access - traversal is O(n), use std::next or std::advance
◆ list::sort() uses merge sort - std::sort requires random access iterators
◆ Cache misses dominate performance - scattered nodes prevent CPU prefetching
◆ 16 bytes pointer overhead per element - makes list inefficient for small types
◆ size() is O(1) since C++11 - maintained counter updated on insert/erase
◆ Default to vector not list - list is specialized, vector wins most real-world cases
