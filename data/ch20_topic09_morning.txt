TOPIC: RAII File Descriptor Wrapper

Automatic resource management for file descriptors using RAII principles

What this carousel covers:
→ RAII pattern: resource lifetime bound to object lifetime with automatic cleanup
→ FileDescriptor class with constructor acquiring FD and destructor releasing it
→ Rule of Five implementation: deleted copy operations, implemented move semantics
→ Exception-safe file operations with read() and write() wrappers
→ Safe file reading with 1024-byte buffer and automatic cleanup on scope exit
→ Pipe communication using RAII to manage two file descriptors (read/write ends)
→ Kernel pipe buffer of typically 64KB for write operations
→ Double close protection using fd_ != -1 check before calling close()

Key insights:
◆ Operating systems limit file descriptors to typically 1024 per process - leaks accumulate and cause "too many open files" errors
◆ Traditional C-style open()/close() is prone to leaks if close() is forgotten or exceptions occur
◆ RAII ensures file descriptor cannot leak even if exception thrown after opening - destructor runs during stack unwinding
◆ Copy constructor deleted prevents double close bug where two owners would close() same FD and crash
◆ Move semantics transfer ownership safely - moved-from object sets fd_ to -1
◆ Unix pipes create two file descriptors consuming from limited pool - explicit close of unused ends prevents resource exhaustion
◆ Closing pipe write end signals EOF to readers for clean shutdown
◆ Rule of Five: defining destructor requires defining all five special member functions (copy ctor/assignment, move ctor/assignment)
