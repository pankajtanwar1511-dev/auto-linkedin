TOPIC: Thread-Safe Singleton - Meyer's & Double-Checked Locking

Ensuring only one instance exists globally in concurrent environments

What this carousel covers:
→ Classic Singleton race condition where two threads can create two instances
→ Meyer's Singleton using C++11 static local variables with guaranteed thread-safe initialization
→ Double-Checked Locking Pattern (DCLP) with std::atomic and memory ordering (acquire/release)
→ Call-Once Pattern using std::once_flag for initialization synchronization
→ Static Deinitialization Fiasco when singleton destructors depend on each other

Key insights:
◆ C++11 guarantees static local variables initialized once, thread-safely using compiler-generated std::once_flag equivalent
◆ Meyer's Singleton avoids manual memory management and provides automatic destruction at program exit
◆ DCLP optimizes by avoiding lock on every call - only locks during first initialization with double-check
◆ Memory ordering matters: use memory_order_acquire for load, memory_order_release for store, memory_order_relaxed inside lock
◆ Function-local statics solve static initialization order fiasco - initialized on first use, not at program start
◆ Mutex contention in FileLogger can bottleneck with frequent logging - consider buffering for high-throughput apps
◆ Flushing after each write trades performance for crash-resistant logging (logs survive crashes)
