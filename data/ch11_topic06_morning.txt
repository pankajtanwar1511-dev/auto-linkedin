TOPIC: Async, Promise, and Future - Asynchronous Task Management

Master C++11's async programming primitives for clean concurrent code

What this carousel covers:
→ std::future as receiver/consumer handle that blocks on get() until results ready
→ std::promise as sender/producer write-once channel for values or exceptions
→ std::async as task launcher combining thread creation with future generation
→ Launch policies: async (guaranteed parallel) vs deferred (lazy evaluation) vs dangerous default
→ std::shared_future for broadcasting results to multiple consumers non-destructively
→ std::packaged_task for thread pool implementations separating task creation from execution
→ Timeout operations with wait_for() and wait_until() returning future_status
→ Automatic exception propagation across threads via stored exception_ptr
→ Real-world autonomous driving sensor fusion with deadline-based waiting patterns

Key insights:
◆ Future destructor from std::async with launch::async blocks until completion - prevents detached threads but causes surprising latency
◆ Always specify explicit launch policy - default (async|deferred) is implementation-defined and usually deferred, defeating asynchronous purpose
◆ Promise-future pairs provide happens-before guarantees - set_value() synchronizes-with get() eliminating need for manual mutexes
◆ Single-use semantics: calling get() twice throws future_error::no_state, calling set_value() twice throws promise_already_satisfied
◆ Broken promise exception when promise destroyed without setting value - prevents indefinite blocking on abandoned promises
◆ Exception timing: exceptions thrown in async tasks stored in shared state, rethrown only when get() called, not immediately
◆ std::packaged_task enables thread pools by wrapping callables with futures - workers execute tasks while submitters retrieve results via futures
◆ Real-time systems require wait_for() with timeouts rather than blocking get() - autonomous vehicle perception has 50-100ms hard deadlines
