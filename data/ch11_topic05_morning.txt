TOPIC: Atomics and Memory Ordering - Lock-Free Programming Fundamentals

Master the art of lock-free concurrent programming with std::atomic and memory ordering.

What this carousel covers:
→ std::atomic operations - lock-free thread-safe operations using hardware atomic instructions
→ Lock-free vs wait-free programming - system progress guarantees and CAS-based algorithms
→ Memory ordering models - relaxed, acquire-release, and sequential consistency semantics
→ Compare-and-swap (CAS) operations - building block for lock-free data structures
→ Happens-before relationships - establishing cross-thread visibility guarantees
→ Atomic operation types - load, store, exchange, fetch_add, and compare_exchange variants
→ Hardware-level atomic instructions - LOCK CMPXCHG, LDREX/STREX, and memory barriers
→ Real-time system applications - autonomous driving sensor fusion with microsecond latency requirements
→ Common pitfalls - ABA problem, spurious CAS failures, false sharing, and memory reclamation
→ Performance implications - relaxed vs seq-cst overhead, cache line bouncing, and optimization strategies

Key insights:
◆ std::atomic provides atomicity and configurable memory ordering without explicit mutex locks
◆ Lock-free algorithms guarantee system-wide progress using CAS loops, even if individual threads retry
◆ Memory ordering controls instruction reordering - relaxed allows maximum reordering, acquire-release synchronizes pairs, seq-cst provides global total order
◆ Acquire-release semantics establish happens-before relationships where all operations before a release store are visible to threads performing an acquire load
◆ Compare-exchange-weak may fail spuriously on ARM architectures - always use in loops for lock-free data structures
◆ The ABA problem occurs when CAS succeeds despite structural changes - solve with tagged pointers, hazard pointers, or epoch-based reclamation
◆ False sharing causes severe performance degradation when independent atomics share cache lines - use alignas(64) to separate hot atomics
◆ Autonomous vehicle sensor fusion requires atomics for frame counters and timestamps to avoid mutex overhead in microsecond-latency pipelines
◆ Never use volatile for multithreading - it provides no atomicity or memory ordering guarantees
◆ Always verify is_lock_free() on target platforms - large atomic types may use hidden mutexes defeating the performance purpose
