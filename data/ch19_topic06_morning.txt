TOPIC: C++20 Modules - Modern Code Organization

Replace the Preprocessor Era

What this carousel covers:
→ Traditional header problems: slow compilation (headers parsed in every TU), order dependency, macro pollution, fragile include guards, transitive includes, ODR violations
→ C++20 Modules solution: compiled once and reused, explicit exports only, no macro leakage, order-independent imports, isolated implementation details, better IDE tooling
→ Module declaration and export: export module syntax, individual exports, export blocks, non-exported internal functions, module interface units
→ Module partitions: split large modules with :partition syntax, primary module re-exports partitions (export import :arithmetic), partition names start with colon
→ Module implementation units: separate interface (.ixx) from implementation (.cpp), module keyword without export for implementation
→ Importing standard library: import std (C++23 future), current C++20 reality requires #include for std library
→ Global module fragment: module; syntax before declaration, allows #include for backward compatibility, bridges headers and modules
→ Private module fragment: module :private; syntax hides implementation completely, helper functions not visible to importers
→ Module linkage rules: module-internal variables, exported variables (module linkage), internal linkage (anonymous namespace), external linkage (extern "C")
→ Migration strategy: create module interface, provide header compatibility with #ifdef __cpp_modules, gradual transition, CMake 3.28+ support

Key insights:
◆ Compilation speed impact: large codebases see 50-90% compile time reduction with modules, clean builds 10-100x faster vs headers, every TU parsing headers eliminated
◆ Macro isolation by design: #define MAX 100 in module doesn't export to importers, prevents preprocessor pollution, use export constexpr instead
◆ Partition organization: export module math:arithmetic for partitions, primary module controls re-exports, users only import primary module, parallel compilation possible
◆ Global module fragment necessity: module; before export module required for #include headers, needed because std library isn't modules yet, macros from included headers don't leak
◆ Private fragment implementation hiding: module :private; completely hides implementation details, ConnectionImpl class not visible to importers, cleaner encapsulation than headers
◆ Template export requirements: templates must be exported if users instantiate them, implementation must be in interface for templates, Container<T> not exported causes instantiation errors
◆ Build system challenges: CMake 3.28+ adds FILE_SET CXX_MODULES support, compiler differences (MSVC/GCC/Clang vary), third-party libraries still use headers
◆ Migration compatibility layer: #ifdef __cpp_modules allows gradual transition, math_compat.h bridges module and header users, eventually remove compatibility headers
◆ Explicit export control: only export module declarations visible, non-exported functions are module-internal, export { } block syntax for multiple declarations
◆ Measured benefits: 50-90% faster builds, no transitive includes, order-independent imports, clean separation of interface and implementation, huge projects become manageable
