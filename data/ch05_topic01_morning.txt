TOPIC: Operator Overloading

Enable custom types to use familiar operator syntax while maintaining type safety and intuitive semantics for expressive, maintainable APIs.

What this carousel covers:
→ Operator overloading mechanism - operators are functions with special names (operator+)
→ What can be overloaded - arithmetic, comparison, subscript, function call, stream I/O
→ What cannot be overloaded - scope resolution (::), member selection (.), ternary (?:), sizeof
→ Member vs non-member functions - left operand determines member function, symmetry needs friends
→ Stream insertion/extraction - must be non-member friends for correct syntax (cout << obj)
→ Compound assignment idiom - implement += first, then define + using +=
→ Comparison operators - implement == and <, derive others (C++20 spaceship fixes this)
→ Prefix vs postfix increment - prefix returns reference, postfix takes dummy int parameter
→ Real-world Complex number class with natural mathematical syntax

Key insights:
◆ Member functions have implicit left operand (a + b → a.operator+(b)), non-members are symmetric
◆ Binary operators should be non-member friends when both operands are user types
◆ Stream operators must be non-member - cout << obj requires operator<<(ostream&, const T&)
◆ Return types matter - arithmetic returns by value, compound assignment returns *this by reference
◆ Overload semantics should match intuition - don't make operator+ perform subtraction
