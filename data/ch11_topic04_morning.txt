TOPIC: Atomics and Memory Ordering - Lock-Free Programming Fundamentals

Understanding lock-free programming through std::atomic operations and memory orderings.

What this carousel covers:
→ std::atomic template class and hardware atomic instructions (LOCK CMPXCHG, LDREX/STREX)
→ Lock-free vs wait-free programming guarantees and system-wide progress
→ Five memory orderings: relaxed, acquire, release, acq_rel, and seq_cst
→ Compare-and-swap (CAS) operations for lock-free data structures
→ Happens-before relationships and acquire-release synchronization
→ Memory reordering prevention and visibility guarantees across threads
→ Hardware-level atomic operations on x86 (LOCK prefix) and ARM (load-exclusive/store-exclusive)
→ Atomic operations: load, store, exchange, fetch_add, compare_exchange_strong/weak
→ Real-world autonomous driving applications with microsecond-level requirements
→ False sharing mitigation using cache-line alignment (alignas(64))

Key insights:
◆ std::atomic provides lock-free operations using CPU atomic instructions, avoiding syscalls and context switches (20-200x faster than mutexes for simple operations)
◆ memory_order_relaxed provides atomicity without ordering constraints - use only for independent counters where cross-thread ordering doesn't matter
◆ Acquire-release semantics establish happens-before relationships: all operations before release are visible to threads performing acquire loads
◆ compare_exchange_weak may fail spuriously (allowed by hardware), always use in loops for lock-free algorithms with retry logic
◆ The ABA problem occurs when values change A→B→A, causing CAS to succeed incorrectly - requires tagged pointers or hazard pointers to solve
◆ False sharing causes cache line bouncing when independent atomics share 64-byte cache lines - pad with alignas(64) for 10-100x performance improvement
◆ Volatile is NOT for threading - provides zero atomicity or memory ordering, only prevents compiler register caching for memory-mapped I/O
◆ Production lock-free code requires memory reclamation strategies (hazard pointers, epoch-based reclamation) to prevent use-after-free bugs
