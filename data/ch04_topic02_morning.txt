TOPIC: Move Semantics and std::move

Transfer resource ownership efficiently by "stealing" internal pointers instead of expensive deep copies - the performance revolution of C++11.

What this carousel covers:
→ Move semantics problem - before C++11, returning large objects forced expensive O(n) copies
→ Resource transfer mechanism - move constructors steal pointers and nullify source in O(1) time
→ std::move is just a cast - converts lvalue to rvalue reference, doesn't move anything itself
→ When moves happen automatically - returning locals, passing temporaries, container reallocation
→ Valid-but-unspecified state - moved-from objects remain destructible and assignable
→ Rule of Five - defining destructor, copy/move constructors, copy/move assignment together
→ Rule of Zero preferred - use smart pointers so compiler generates all operations correctly
→ Common pitfalls - moving from const, returning std::move(local) prevents RVO, self-move bugs
→ Real-world autonomous vehicle trajectory transfer - moving 1M waypoint trajectories efficiently

Key insights:
◆ std::move doesn't move - it's static_cast<T&&> that enables move operations to be called
◆ Move vs copy: O(1) pointer theft vs O(n) element duplication - ~1000× faster for large containers
◆ Moved-from objects are valid but unspecified - safe to destroy or assign, unsafe to read
◆ Never use std::move in return statements - prevents RVO and forces unnecessary move
◆ noexcept on move operations is critical - containers use copies without it for exception safety
