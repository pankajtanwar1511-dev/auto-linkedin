TOPIC: Compile-Time Programming in C++

Moving computation from runtime to compile-time for zero-cost abstractions.

What this carousel covers:
→ constexpr evolution (C++11 through C++20) - from single-expression recursion to full imperative code
→ constexpr vs consteval vs constinit - when to use each for different compile-time guarantees
→ if constexpr for compile-time branch elimination - discarded branches never instantiated
→ static_assert with type traits - enforcing constraints at compile-time
→ Template metaprogramming vs constexpr functions - instantiation depth limits and workarounds
→ Type traits (is_same, is_integral, is_pointer) - compile-time type introspection
→ SFINAE with enable_if - type-dependent function overload resolution
→ Custom type traits with template specialization - creating domain-specific compile-time checks
→ Real-world autonomous vehicle examples - sensor calibration validation, CRC32 hashing, lookup tables
→ Debugging constexpr failures - techniques for identifying non-constant operations

Key insights:
◆ C++11 constexpr is restrictive (single return, no loops) - C++14+ allows loops, local variables, multiple statements
◆ constexpr permits runtime evaluation with non-constant inputs - consteval mandates compile-time (C++20)
◆ if constexpr eliminates dead branches at compile time - invalid code in discarded branches won't cause errors
◆ Template recursion hits depth limits (~512-1024) - constexpr functions have no such limit
◆ constinit guarantees compile-time initialization for statics - prevents initialization order fiasco while allowing mutation
◆ Type traits enable SFINAE for type-based overload resolution - compile-time function selection without runtime checks
◆ Compile-time evaluation has zero runtime cost - results are literal constants baked into binary
◆ static_assert catches invalid configurations before runtime - critical for safety systems (sensor dimensions, buffer sizes)
