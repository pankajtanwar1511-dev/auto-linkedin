TOPIC: Custom Hash Map - Separate Chaining Implementation

Building production-grade hash tables from scratch

What this carousel covers:
→ Separate chaining collision resolution with linked list buckets
→ Dynamic rehashing when load factor α exceeds 0.75 threshold
→ Custom hash function design using std::hash composition
→ O(1) average-case insert/find/erase operations with amortized complexity
→ Complete HashMap implementation with 16 initial buckets and 2× growth strategy
→ Real-world word frequency counter demonstrating operator[] auto-insertion

Key insights:
◆ Load factor α = n/m (elements/buckets) should stay around 0.75 - below this gives low collision probability (~37% per Poisson distribution), above this causes chains to grow longer and performance degrades
◆ Rehashing at 2× bucket size has O(n) cost but amortizes to O(1) per insertion over many operations
◆ Bad hash functions cause O(n) worst-case performance - all keys colliding into one bucket transforms into linear search
◆ References to values become dangling after rehash since the entire bucket vector gets reallocated
◆ operator[] provides std::unordered_map semantics with automatic Value{} (0 for int) initialization on missing keys
◆ Separate chaining avoids clustering issues of open addressing but requires extra memory for list pointers
◆ Custom hash composition: std::hash<std::string>{}(name) ^ (std::hash<int>{}(age) << 1) combines multiple fields
