TOPIC: Copy Constructor vs Assignment Operator and Rule of Three/Five/Zero

Resource management fundamentals that prevent memory leaks, double-deletion, and undefined behavior.

What this carousel covers:
→ Copy constructor vs copy assignment operator - when each is called, key differences
→ Self-assignment safety - why "if (this != &other)" prevents data corruption
→ Rule of Three (C++98) - destructor, copy constructor, copy assignment together
→ Rule of Five (C++11+) - adds move constructor and move assignment for performance
→ Rule of Zero (modern C++) - prefer smart pointers and RAII over manual resource management
→ Copy-and-swap idiom - strong exception safety and automatic self-assignment protection
→ Shallow vs deep copy - default bitwise copy causes double-deletion crashes
→ noexcept on move operations - critical for optimal container performance
→ Copy elision (RVO/NRVO) - C++17 mandatory optimization eliminates copy/move
→ Compiler-generated special member function rules - what gets suppressed when
→ Autonomous vehicle sensor data buffer example - Rule of Five implementation

Key insights:
◆ Copy constructor creates new objects during initialization; assignment modifies existing objects requiring cleanup
◆ Default shallow copy of raw pointers causes double-deletion - multiple objects delete the same memory
◆ Rule of Zero is preferred: use smart pointers to eliminate manual resource management entirely
◆ Mark move operations noexcept so containers use moves instead of copies during reallocation
◆ Never use std::move on return values - it prevents RVO/NRVO optimization
