TOPIC: C++ Low-Level Internals and Common Pitfalls

Understanding memory layout, undefined behavior, and tricky C++ semantics

What this carousel covers:
→ Object memory layout with vtables and vptr mechanisms for runtime polymorphism
→ Virtual dispatch implementation: how virtual function calls translate to vtable lookups
→ Alignment and padding optimization: ordering struct members to minimize memory waste
→ Cache line alignment (alignas) to prevent false sharing in multithreaded code
→ Lifetime management: dangling references, temporary lifetime extension rules with const refs
→ Const correctness: const pointers variations and mutable keyword for logical constness
→ Undefined behavior categories: null pointer deref, signed overflow, use-after-free
→ Copy semantics: shallow vs deep copy, Rule of Three/Five, and why smart pointers solve these problems
→ Virtual inheritance and the diamond problem: solving ambiguity with shared base classes
→ Virtual function behavior in constructors/destructors: why vptr points to current class

Key insights:
◆ Objects with virtual functions contain 8-byte vptr at the beginning, adding memory overhead but enabling dynamic dispatch
◆ Reordering struct members from largest to smallest type can reduce size by 33% (e.g., 12 bytes to 8 bytes)
◆ Object slicing when passing polymorphic types by value loses derived parts and vptr - always use pointers/references
◆ Const reference lifetime extension only works for temporaries in same scope, not when returning from functions
◆ Signed integer overflow is undefined behavior; unsigned overflow wraps around (well-defined)
◆ Default copy constructor does shallow copy for pointers, causing double-delete - implement Rule of Three or use smart pointers
◆ Virtual inheritance adds virtual base pointer overhead (~8 bytes) but solves diamond problem ambiguity
◆ During construction/destruction, vptr points to current class vtable to prevent accessing uninitialized derived members
