TOPIC: C++17 Parallel Algorithms - Execution Policies and Performance

Transform single-threaded code into multi-core powerhouses with one parameter

What this carousel covers:
→ Four execution policies: seq, par, par_unseq, unseq - automatic parallelization without manual threading
→ 4-8x speedup on suitable workloads: 1920x1080 image processing drops from 45ms to 8ms (5.6x faster)
→ Thread safety requirements: par allows locks/allocations, par_unseq requires pure computations
→ Performance thresholds: beneficial above 10,000 elements for sort, 100,000 for reduce
→ Real benchmarks: 10M element sort improves from 1450ms to 250ms (5.8x speedup)
→ LiDAR point cloud processing: 100,000 points filtered and normalized in parallel
→ Critical pitfalls: data races with shared state, locks in vectorized code cause UB
→ Small datasets: 100 elements parallel sort runs 25x SLOWER (50μs vs 2μs) due to overhead

Key insights:
◆ Execution policies are permission not commands - implementation may execute sequentially if parallelization isn't beneficial
◆ std::reduce replaces std::accumulate for parallel sum - uses per-thread accumulators to avoid data races
◆ par_unseq enables SIMD vectorization but prohibits locks, allocations, and exceptions
◆ Floating-point results may differ: parallel execution reorders operations, exposing non-associativity (1e20 + 1.0 + (-1e20) = different results)
◆ transform_reduce fuses map-reduce patterns: dot product without intermediate storage, ideal for ML inference
◆ Autonomous vehicle perception: camera frame processing (2M pixels) achieves 5.6x speedup with par_unseq
◆ Exception handling: one thread throws terminates all parallel work, multiple throws call std::terminate()
◆ Iterator efficiency: random-access preferred for work distribution, forward/bidirectional may not parallelize
