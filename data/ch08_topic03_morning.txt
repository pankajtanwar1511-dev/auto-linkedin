TOPIC: std::deque and std::forward_list - Alternative Sequence Containers

Double-ended queue and singly-linked list designed for specialized performance needs.

What this carousel covers:
→ Deque's chunked architecture - dynamic array of fixed-size chunks (512B-4KB) with central map enabling O(1) operations at both ends
→ Random access via chunk calculation - chunk_number = index/CHUNK_SIZE, chunk_offset = index%CHUNK_SIZE for O(1) indexed access
→ Forward_list's minimal design - eliminates prev pointer for 50% memory savings (8B vs 16B overhead per node)
→ The before_begin() pattern - sentinel iterator enabling front operations following forward_list's consistent "after-based" API
→ Iterator invalidation rules - Deque: complex (end ops preserve, middle invalidates all); forward_list: excellent stability
→ Operation asymmetry - Deque excels at double-ended access with random access capability; forward_list lacks back operations entirely
→ Memory layout trade-offs - Deque non-contiguous (no C API compatibility); forward_list scattered nodes (poor cache locality)

Key insights:
◆ Deque provides the unique combination of O(1) random access AND efficient double-ended operations - the only STL container offering both capabilities
◆ Choose deque for queue implementations where push_back (enqueue) and pop_front (dequeue) must both be O(1) - vector would be O(n) for pop_front
◆ Forward_list uses half the pointer overhead of list (8 bytes vs 16 bytes per node on 64-bit systems), ideal for memory-critical embedded systems
◆ Sliding window algorithms benefit from deque's double-ended removal capability - maintain monotonic queue patterns in O(1) per element
◆ Work-stealing queues leverage deque's architecture - owner operates one end, thieves steal from opposite end with minimal contention
◆ Forward_list's lack of size() is deliberate - maintaining a size counter would add overhead to splice_after operations, violating minimal-overhead design philosophy
◆ Deque's chunked storage provides better cache locality than list but worse than vector - intra-chunk access is fast, cross-chunk jumps cause cache misses
◆ All forward_list insertion/deletion uses "after" pattern (insert_after, erase_after) reflecting singly-linked structure requiring access to preceding node
