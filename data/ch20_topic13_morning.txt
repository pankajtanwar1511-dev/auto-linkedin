TOPIC: Thread-Safe Asynchronous Logger

Production-grade logging system with background thread and queue pattern

What this carousel covers:
→ Producer-consumer architecture: multiple threads push to queue, background thread writes to disk
→ Thread-safe queueing with mutex + condition variable synchronization
→ Performance comparison: asynchronous ~1μs vs synchronous ~1ms (1000× faster)
→ Log level filtering: DEBUG, INFO, WARN, ERROR with configurable minimum level
→ Timestamping with std::chrono::system_clock formatted as "YYYY-MM-DD HH:MM:SS"
→ RAII guarantee: destructor flushes all pending messages before program exits
→ Multi-threaded logging example: 4 worker threads × 10 messages with 10ms delays
→ Edge cases: destruction with pending messages and disk full scenarios

Key insights:
◆ Asynchronous logging is critical for high-throughput servers: 1000 requests/second × 5 logs each × 1ms disk write = 5 seconds blocked per second (impossible with synchronous logging)
◆ Worker threads spend only ~1μs in log() vs ~1ms for synchronous disk write - 1000× performance improvement
◆ Real-world applications: web servers (request logging without slowing responses), game servers (60+ FPS logging without frame drops), databases (query logging without impacting transactions)
◆ Queue allows burst handling: temporary high log volume doesn't block application threads
◆ Background thread batches writes and optimizes disk I/O patterns
◆ Bounded queue solutions for preventing OOM: block when full, drop oldest (circular overwrite), or drop newest (skip when full)
◆ Implementation uses std::queue + std::mutex + std::condition_variable for thread-safe producer-consumer pattern
