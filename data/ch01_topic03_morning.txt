TOPIC: Pure Virtual Functions and Abstract Base Classes

Interface contracts and dependency inversion for flexible, testable architectures.

What this carousel covers:
→ Pure virtual functions with = 0 syntax - forcing implementation contracts
→ Abstract base classes (ABCs) - cannot instantiate, only use via pointers/references
→ Pure virtuals CAN have bodies - providing optional shared logic
→ Pure virtual destructors MUST have definitions - linker error otherwise
→ Calling pure virtuals in constructors - undefined behavior and crashes
→ Intermediate abstract classes - partial implementations in inheritance hierarchies
→ Function hiding vs overriding pure virtuals - common mistake with override keyword
→ Template Method pattern - algorithm structure with pure virtual hooks
→ Multiple inheritance with pure interfaces - safe when stateless
→ Autonomous vehicle motion planning example - A*, Hybrid A*, RRT planners

Key insights:
◆ Pure virtual = "must override," not "cannot have body"
◆ Interfaces enable dependency inversion - high-level code depends on abstractions, not concrete implementations
◆ Abstract classes can have constructors, data members, and concrete methods
◆ Use override keyword to catch signature mismatches
