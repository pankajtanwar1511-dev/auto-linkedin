TOPIC: CRTP (Curiously Recurring Template Pattern)

Achieve polymorphic behavior with zero runtime overhead through static dispatch

What this carousel covers:
→ The "curious recursion" where derived class passes itself as template parameter to base
→ Static downcast mechanism using static_cast<Derived*>(this) for compile-time dispatch
→ Zero overhead polymorphism - fully inlined calls without vtable lookups
→ Performance gains: 3.3x faster than virtual functions (15ms vs 50ms for 1M calls)
→ Real-world autonomous vehicle use cases: sensor fusion loops at 1kHz, control algorithms at 10kHz
→ Policy-based design for compile-time algorithm selection (Kalman filter variants, path planning)
→ Mixin pattern for composing behaviors without conflicts (logging, caching, metrics)
→ Edge cases: template name hiding, infinite recursion traps, incomplete type issues with sizeof()
→ Multiple CRTP inheritance patterns and ambiguity resolution strategies
→ When to use CRTP vs virtual functions: decision matrix for performance-critical code

Key insights:
◆ CRTP provides compile-time polymorphism through template instantiation - compiler knows exact derived type and can fully inline calls
◆ Performance critical: 15-30% faster for control algorithms, 5-20% faster for sensor fusion loops - no vtable overhead
◆ Two-phase lookup requires this-> or explicit qualification when accessing base class members in template contexts
◆ Avoid infinite recursion by using different method names (interface() calls interfaceImpl())
◆ Cannot use sizeof(Derived) in base constructors - derived type is incomplete during base instantiation
◆ CRTP does NOT provide runtime polymorphism - use template functions, not base pointers, for generic processing
◆ Hybrid approach combines virtual functions for runtime flexibility with CRTP for performance-critical hot paths
◆ Perfect for sensor processing, state estimators, and control loops where type is known at compile-time
