TOPIC: Layered Architecture Pattern

Organizing code into horizontal layers with strict dependency rules

What this carousel covers:
→ Classic 3-Tier Architecture (Presentation → Business → Data)
→ Layer responsibilities: UI displays, Business validates, Data persists
→ Complete User registration flow with age >= 18 validation
→ Business rules: email format check, uniqueness validation
→ N-Tier Architecture with API Gateway layer for REST endpoints
→ E-Commerce system: order processing with $20 minimum order rule
→ Stock validation and price calculation workflow

Key insights:
◆ Dependency Rule: Upper layers depend on lower layers, NEVER reverse (Presentation → Business → Data, not Data → Business)
◆ Layer violation example: UI directly querying database skips business rules - always use adjacent layers
◆ Repository Pattern abstractions: IUserRepository interface enables MySQLUserRepository, PostgreSQLUserRepository, InMemoryUserRepository swapping
◆ Testing strategy: Mock repositories for unit tests eliminate database dependency (InMemoryUserRepository for tests)
◆ Circular dependency solution: Use callbacks/events instead of lower layer calling upper layer (EventBus publishes UserSavedEvent)
◆ Performance overhead: 7 layers add 5ms function call latency (UI → Controller → API Gateway → Service → Domain → Repository → Database = 15ms total)
◆ God Class problem: Split 5000-line UserService into UserRegistrationService, UserAuthenticationService, UserProfileService
◆ Cross-cutting concerns: LoggingDecorator wraps services to add logging without code duplication
