TOPIC: Multi-Client Server Patterns - Scaling Beyond One Connection

Building servers that handle thousands of concurrent clients efficiently

What this carousel covers:
→ Fork-based servers: Process per client with complete isolation (100-1000 max clients)
→ Thread-based servers: Lightweight threads with shared memory (1000-10000 max clients)
→ Thread pool pattern: Pre-created worker threads for optimal CPU utilization
→ Performance comparison: Thread creation ~2-3x faster than fork() at scale
→ Zombie process handling: SIGCHLD handlers and waitpid() with WNOHANG
→ Race conditions: Protecting shared data with mutexes and atomic operations
→ Resource limits: Process costs ~10MB, thread costs ~8KB, fork time 1-2ms vs thread 10μs

Key insights:
◆ Single-client bottleneck: If handle_client() takes 10 seconds, 1000 vehicles wait over 2 hours for map updates
◆ Fork overhead: Separate address space per process, limited to ~1000-10000 max due to OS constraints
◆ Thread scalability: Can handle 1000-10000 concurrent clients with shared memory and faster creation
◆ Thread pool efficiency: Pre-created threads eliminate creation overhead, optimal thread count = CPU cores (CPU-bound) or 2-4x cores (I/O-bound)
◆ Zombie accumulation: Child processes become defunct if parent doesn't call waitpid(), consuming process table entries until fork() fails with EAGAIN
◆ Race condition anatomy: client_count++ expands to 3 operations (read, increment, write), causing lost updates when threads interleave
◆ Benchmark results: 10 clients: fork 250ms vs threads 150ms | 100 clients: fork 2500ms vs threads 1200ms | 1000 clients: threads 11000ms (fork impractical)
◆ Memory comparison: Fork ~10MB per client, threads ~8KB per client, thread pool fixed overhead regardless of client count
◆ Crash isolation: Fork provides complete isolation (child crash doesn't affect parent), threads share memory (one crash kills entire process)
◆ Production pattern: Thread pools used in nginx, Apache, database connection pools for bounded resources and predictable performance
