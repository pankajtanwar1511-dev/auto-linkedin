TOPIC: Observer Pattern (Publish-Subscribe)

Behavioral design pattern defining one-to-many dependencies where subject automatically notifies dependent observers of state changes.

What this carousel covers:
→ Core components: Subject (observable), Observer (dependent), and their responsibilities in one-to-many relationships
→ Push vs Pull models: Subject sending data vs observers querying for needed information
→ Smart pointer implementation using weak_ptr to prevent circular references and automatic observer cleanup
→ Thread-safe observers with mutex protection, snapshot copying, and deadlock prevention strategies
→ Signal/Slot pattern with std::function for type-safe, flexible callback systems
→ Notification ordering with priority-based execution and multi-phase notifications
→ Exception safety: Isolating observer failures to ensure all observers execute even if some throw
→ Autonomous vehicle sensor fusion: Real-world example with perception, localization, path planning, and safety systems
→ Asynchronous observers with event queues for non-blocking notifications in real-time systems
→ Common pitfalls: Circular references, observer deletion during notification, notification storms, and race conditions

Key insights:
◆ Use weak_ptr for observers to break circular reference cycles (subject holds weak_ptr, observers can hold weak_ptr to subject)
◆ Copy observer list before notifying to prevent iterator invalidation when observers detach themselves during callbacks
◆ Release mutex before invoking observer callbacks to avoid deadlocks (copy observers under lock, notify without lock)
◆ Recursion guards prevent notification storms when observers modify subject state (use notifying flag or deferred update queue)
◆ Push model sends all data (simpler but wasteful), pull model provides query interface (efficient, lower coupling)
◆ Priority-based notification ensures critical observers (emergency braking) execute before logging/analytics
◆ Catch exceptions in notification loop to ensure one failing observer doesn't prevent others from executing
◆ Asynchronous observers with background thread processing prevent slow callbacks from blocking subject
