TOPIC: Fixed-Size Memory Pool Allocator

High-Performance Memory Management

What this carousel covers:
→ Standard heap allocation issues: malloc/new takes ~100-500ns vs memory pool ~5-20ns (10-50x faster)
→ Memory pool design with pre-allocated contiguous blocks and intrusive free list management
→ O(1) allocation/deallocation using linked list operations with zero fragmentation
→ Alignment requirements: alignof(int)=4, alignof(double)=8, alignof(std::max_align_t)=16
→ Production implementation using union Block with proper aligned_alloc
→ Critical edge cases: alignment issues, double-free detection, pool exhaustion, destructor management
→ Cache line false sharing: padding to 64-byte cache line size for atomic counters
→ Real benchmark: 100,000 allocations - standard ~45ms vs pool ~3ms (15x speedup)
→ Thread-local pools for multithreading: 8 threads with shared pool+mutex ~500ms vs thread-local pools ~70ms (7x faster)
→ RAII wrapper (PoolPtr) for exception safety and automatic cleanup

Key insights:
◆ Memory pool eliminates system call overhead after initialization - no malloc/brk/mmap calls during runtime
◆ Intrusive free list stores next pointers inside free blocks themselves - zero overhead when allocated
◆ Cache locality advantage: sequential pool allocations enable hardware prefetcher, reducing access from ~100ns (cache miss) to ~5ns (cache hit) - 20x faster
◆ malloc overhead is ~16-32 bytes per allocation vs pool's ~8 bytes per block (free list pointer only)
◆ Thread-local pools achieve perfect scalability: no locks, no contention, linear scaling with thread count
◆ External fragmentation wastes memory with standard allocation but fixed-size blocks in pools eliminate fragmentation completely
◆ std::aligned_alloc ensures proper alignment: misaligned access causes undefined behavior or crashes on ARM/older x86
◆ Exception safety requires RAII: without it, exceptions cause memory leaks as pool.destroy() never executes
◆ For 10,000 node linked list: pool-based allocation is 20-50x faster than new/delete and uses 2-3x less memory
◆ Real-world applications: game engines (particle systems), network servers (connection objects), databases (B-tree nodes), trading systems (microsecond to nanosecond latency reduction)
