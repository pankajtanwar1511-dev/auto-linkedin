TOPIC: Strategy Pattern (Policy-Based Design)

Defining families of algorithms and making them interchangeable at runtime or compile-time.

What this carousel covers:
→ Core components: Strategy interface, Concrete Strategies, Context class, and Client selector
→ Runtime vs compile-time strategy selection (virtual functions vs templates)
→ Five C++ implementation approaches: virtual functions, function pointers, std::function, functors, and std::variant
→ Real-world autonomous vehicle path planning with dynamic strategy switching
→ Performance comparison: 3-5ns overhead for virtual vs zero overhead for templates
→ Critical edge cases: context data coupling, thread safety for stateful strategies, and ownership management
→ Strategy selection with Factory pattern to encapsulate complex decision logic
→ Preventing mid-execution strategy changes to maintain state consistency

Key insights:
◆ Strategy pattern encapsulates algorithms behind common interface, enabling runtime algorithm swapping without modifying context
◆ Four core guarantees: interchangeability (same interface), encapsulation (isolated logic), Open/Closed Principle (add strategies without context changes), composition over inheritance
◆ Runtime strategy (virtual functions) provides flexibility at 3-5ns cost; compile-time strategy (templates/policies) gives zero overhead but fixed at compile-time
◆ Use std::unique_ptr for exclusive ownership, shared_ptr for shared strategies - never raw pointers to avoid dangling references
◆ Stateful strategies with cache/counters need thread-safety (mutex) or use thread_local storage for per-thread state
◆ Avoid circular dependencies: pass needed data as parameters or define minimal context interface instead of passing entire context
◆ Prevent nullptr dereference by requiring strategy in constructor, providing default strategy, or using Null Object pattern
◆ Factory pattern encapsulates strategy selection logic based on conditions (file type, size, configuration) keeping client code clean
