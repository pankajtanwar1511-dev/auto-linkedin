TOPIC: C++17 Standard Library - optional, variant, any, string_view, filesystem

Type-safe alternatives to error-prone patterns with zero-overhead abstractions

What this carousel covers:
→ std::optional<T>: Stack-allocated wrapper for "may not have value" semantics, replacing pointers and sentinel values
→ std::variant<Ts...>: Type-safe discriminated union holding one of fixed types with automatic lifetime management
→ std::any: Runtime type erasure storing any type with RTTI, replacing void* with type safety
→ std::string_view: Non-owning 16-byte string reference enabling O(1) substring operations without allocation
→ std::filesystem: Platform-independent file operations with path manipulation and directory traversal

Key insights:
◆ optional operations: *opt is fastest but UB if empty, opt.value() throws std::bad_optional_access, opt.value_or(default) is safe with no exceptions
◆ variant size: max(sizeof(Ts...)) + discriminator (typically 1-8 bytes overhead) with zero-cost type dispatch via std::visit
◆ variant vs inheritance: Stack storage vs heap, closed type set vs open, compiler-checked exhaustiveness vs manual checking
◆ any overhead: Fixed 16-32 bytes size with small buffer optimization, may allocate for large types, strips const/volatile qualifiers
◆ string_view is O(1) for construction, copy, and substring vs O(n) for std::string on same operations
◆ string_view lifetime safety: Safe for function parameters, dangerous for return values, unsafe for temporary strings
◆ filesystem path operations: Use / operator for portable path concatenation, fs::canonical() resolves symlinks and . and ..
◆ filesystem queries: fs::exists(), fs::file_size() returns uintmax_t, fs::last_write_time() gets modification timestamp
◆ filesystem iteration: directory_iterator for single level, recursive_directory_iterator for full tree traversal
◆ Error handling: Throwing versions for exceptional errors, error_code overloads for expected failures in production code
