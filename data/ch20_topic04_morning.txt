TOPIC: Thread Pool from Scratch

Reusing worker threads for efficient parallel task execution

What this carousel covers:
→ Thread pool architecture: pre-created worker threads executing tasks from shared queue
→ Avoiding thread creation overhead: ~50-100μs per thread creation, ~1MB stack per thread
→ Design decisions: std::function for tasks, bounded queue, fixed pool size, graceful/immediate shutdown
→ submit() for fire-and-forget tasks, submit_with_result() returning std::future for results
→ Exception handling in worker loop: catch all exceptions to prevent worker thread death
→ Worker loop: blocking wait on condition variable, pop task, execute outside lock
→ Shutdown strategies: graceful (finish queued tasks) vs immediate (discard queued)
→ Task synchronization using std::packaged_task connecting callable to std::future
→ Thread pool vs std::async: thread reuse (8 threads for 1000 tasks) vs potential 1000 thread creations

Key insights:
◆ Thread creation overhead: Creating 1000 threads = 50-100ms + ~1GB memory; thread pool with 8 threads = <1ms + ~8MB memory
◆ Optimal thread count: CPU-bound tasks use std::thread::hardware_concurrency(); I/O-bound use formula: Cores × (1 + Wait_Time / Compute_Time)
◆ Worker loop implementation: condition_.wait() with predicate (!tasks_.empty() || stop_), execute task outside lock to prevent contention
◆ Graceful shutdown: set stop_ flag, notify_all() to wake workers, join() all threads to wait for queued tasks completion
◆ Immediate shutdown: set stop_ and terminate_ flags, clear task queue with std::swap(), notify and join threads
◆ Exception safety: try-catch in worker loop prevents thread death; packaged_task stores exceptions in future, rethrown on get()
◆ Parallel map-reduce pattern: 10 tasks computing sum of 1000-element ranges, futures collect results, main thread aggregates
◆ Task granularity threshold: parallelism beneficial when task duration > ~10-50μs; 1μs tasks are 0.07× slower, 100μs tasks give 6.67× speedup on 8 cores
◆ Matrix multiplication parallelization: row-wise distribution, std::ref avoids copying matrices, scalability for large matrices (1000×1000) gives ~N× speedup on N cores
◆ Work stealing optimization: each thread has local deque, pops LIFO (cache-friendly), steals FIFO from other threads when idle (used by Intel TBB)
